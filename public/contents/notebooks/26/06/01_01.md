Saya masih ingat pertama kali mencoba Deno. Tahun 2020, saat versi 1.0 baru dirilis, rasanya seperti melihat prototype menarik yang belum siap dipakai serius. Tidak ada kompatibilitas npm, ekosistem tools masih kosong, dan dokumentasi terasa eksperimental. Saya mencoba beberapa hari, lalu kembali ke Node.js — ekosistemnya sudah lengkap, stack overflow penuh jawaban, dan npm punya package untuk segala kebutuhan.

Lima tahun kemudian, Deno 2.8 baru saja dirilis pada 22 Mei 2026, dan posisinya sudah sangat berbeda. Bukan lagi runtime alternatif yang lucu untuk dicoba akhir pekan — ia sudah menjadi pesaing nyata di lanskap JavaScript runtime. Dan bagi developer web yang sudah bertahun-tahun tinggal di ekosistem Node.js, saatnya mulai memperhatikan.

## Bagaimana Kita Sampai di Sini

Untuk memahami mengapa Deno 2.8 penting, perlu melihat konteks besarnya. Ryan Dahl menciptakan Node.js pada 2009, mengubah cara dunia membangun aplikasi server-side dengan JavaScript. Tapi ia meninggalkan proyek tersebut pada 2012, dan Node.js berkembang tanpa dia (Wikipedia, "Ryan Dahl").

Pada 2018, di JSConf EU, Dahl kembali tampil dengan presentasi yang kini legendaris: "10 Things I Regret About Node.js". Ia mengakui keputusan desain yang ia anggap keliru — tidak menggunakan Promise di API, desain `node_modules` yang menyebabkan dependency tree raksasa, ketergantungan pada `package.json`, dan file extension resolution yang membingungkan (Dahl, 2018).

Dari penyesalan itu lahirlah Deno: runtime yang dibangun dengan V8 dan Rust (bukan C++ seperti Node.js), mendukung TypeScript secara native tanpa konfigurasi, dan mengutamakan keamanan dengan permission-based sandbox. Nama "Deno" sendiri adalah anagram dari "Node."

Tapi jalan Deno tidak mudah. Saat versi 1.0 dirilis Mei 2020, mereka harus bersaing tidak hanya dengan Node.js yang sudah berusia 11 tahun, tapi juga Bun yang muncul pada 2021 dengan janji kecepatan luar biasa menggunakan JavaScriptCore — engine yang mempowered Safari. Bun mengalami perubahan dramatis sendiri: diakuisisi Anthropic pada Desember 2025 untuk mendukung Claude Code, dan pada Mei 2026 Jarred Sumner mulai menulis ulang seluruh codebase Bun dari Zig ke Rust menggunakan AI tools, mencakup lebih dari satu juta baris kode (Wikipedia, "Bun (software)").

Lanskap runtime JavaScript tahun 2026 jauh lebih crowded daripada lima tahun lalu.

## Deno 2.8: Apa yang Berubah

Rilis 2.8 disebut sebagai "minor release terbesar" yang pernah dibuat tim Deno. Lebih dari 500 commit sejak versi 2.7, menyentuh hampir semua aspek runtime. Dan melihat daftar perubahannya, klaim itu masuk akal.

### Kompatibilitas Node.js: Lompatan Kuantitatif

Angka yang paling mencolok: pass rate Deno terhadap test suite resmi Node.js melonjak dari sekitar 42% di versi 2.7 menjadi **76.4%** di versi 2.8 — itu 3.405 dari 4.457 test Node.js yang berhasil dilewati. Sebagai pembanding, Bun 1.3.14 pada test suite yang sama hanya mencapai 40.6% (Iwańczuk, 2026).

Angka ini bisa dilacak secara real-time di `node-test-viewer.deno.dev`, dimana Deno menampilkan progress mereka terhadap setiap module Node.js. Ini adalah pendekatan engineering yang saya apresiasi — transparan, terukur, dan bisa diverifikasi siapa saja.

Bagi developer yang sudah investasi di ekosistem npm, angka ini penting. Selama ini hambatan terbesar adopsi Deno adalah pertanyaan sederhana: "apakah package npm favorit saya bisa jalan di Deno?" Dengan tiga perempat test suite Node.js lolos, jawabannya makin sering "ya." Banyak module Node.js built-in sekarang juga di-lazy-load, sehingga program yang tidak menggunakannya start lebih cepat tanpa overhead.

### Enam Subcommand Baru

Deno 2.8 menambahkan enam perintah baru yang menyentuh hampir setiap aspek workflow development:

**`deno ci`** — Satu perintah untuk CI/Docker yang melakukan instalasi frozen dari lockfile. Error langsung jika lockfile hilang atau tidak sinkron. Menggantikan kebutuhan menghafal kombinasi flag yang tepat untuk `deno install` di pipeline CI.

**`deno pack`** — Menggabungkan fungsi TypeScript compiler dan `npm pack` dalam satu langkah. Build proyeok Deno atau JSR menjadi tarball yang siap publish ke npm, lengkap dengan TypeScript yang sudah ditranspilasi ke JavaScript dan file `.d.ts` declaration. Specifier juga di-rewrite otomatis — `jsr:@std/path` menjadi `@jsr/std__path`, `npm:express@4` menjadi `express`.

**`deno transpile`** — Strip type dari TypeScript/JSX/TSX ke JavaScript murni tanpa bundling atau module rewriting. Berguna saat butuh artifact JS-only untuk runtime yang tidak mendukung TypeScript native.

**`deno audit fix`** — Otomatis upgrade package yang rentan ke versi patched terdekat. Jika butuh major upgrade, ia menandai secara terpisah. Ini menjawab kebutuhan security yang selama ini harus ditangani tool terpisah seperti `npm audit fix`.

**`deno bump-version`** — Update versi di `deno.json` atau `package.json` sesuai semantic versioning. Dalam workspace monorepo, bisa otomatis bump semua member package sekaligus dan rewrite cross-package references agar tetap sinkron.

**`deno why`** — Menjelajahi dependency tree untuk menjelaskan mengapa suatu package terinstall, bekerja dengan baik npm maupun JSR dependency — setara dengan `npm explain` atau `pnpm why`.

### Performa: Angka yang Berbicara

Dari sisi performa, Deno 2.8 membawa peningkatan yang substansial di beberapa area kritis:

- **Cold npm install 3.66x lebih cepat** — dari 3.319ms turun ke 906ms. Peningkatan ini didapat dari tiga perubahan teknis: abbreviated packument (menggunakan metadata yang lebih kecil dari npm registry), parallel resolution (dependency tree di-resolve secara paralel, bukan sekuensial), dan decompression yang di-offload dari async event loop ke blocking thread pool.
- **node:http throughput 2.21x lebih cepat** — dari 8.339 req/s naik ke 18.431 req/s. P99 latency turun sekitar 40%.
- **node:buffer base64 3.07x lebih cepat** berkat switch ke library `simdutf` untuk encoding/decoding.
- **node:crypto scrypt 2.12x lebih cepat** dan `node:fs` recursive `cpSync` 1.49x lebih cepat.
- **RSS bloat berkurang 3-5x di Linux** berkat `malloc_trim` setelah module loading dan saat Worker terminate — masalah yang sangat terasa saat loading codebase TypeScript besar.

Yang perlu dipahami: angka-angka ini bukan benchmark sintetis yang hanya relevan di laboratorium. Cold install, HTTP throughput, dan memory usage adalah metrik yang dirasakan developer setiap hari, di setiap `npm install`, di setiap request yang dilayani server.

### `import defer`: Optimasi Startup yang Elegant

Deno 2.8 mendukung proposal TC39 `import defer` — kemampuan memuat dan mem-parsing modul tanpa menjalankan top-level code-nya. Modul hanya di-evaluate saat export-nya pertama kali diakses.

Contoh konkret: bayangkan aplikasi yang bisa membaca format PNG dan JPEG, tapi di setiap run hanya menggunakan salah satu. Dengan `import defer`, kedua decoder dimuat di memori sebagai parsed module, tapi hanya yang dipilih yang benar-benar di-evaluate. Ini bisa memangkas startup time secara signifikan untuk aplikasi dengan banyak dependency yang jarang digunakan — scenario yang sangat umum di proyek web modern yang mengimpor puluhan library.

Fitur ini juga bekerja di file `.ts` dan `.tsx` tanpa setup tambahan, dan `deno check` serta LSP sudah mendukung syntax-nya.

### TypeScript 6.0.3 dan Node Types Default

Deno membundel TypeScript 6.0.3 — versi transisi yang menyiapkan perubahan breaking sebelum TypeScript 7.0 yang akan ditulis ulang dalam native port. Yang lebih praktis: `deno check` dan LSP sekarang meng-include `lib.node` secara default, artinya `Buffer`, `process`, `NodeJS.Timeout` dan tipe-tipe Node.js lainnya langsung available tanpa konfigurasi. Ini menyelesaikan masalah yang selama ini membuat type-checking saat mengimpor npm package di Deno terasa cumbersome.

### Debugging yang Setara Browser

Chrome DevTools sekarang bisa memantau network traffic Deno — setiap `fetch()`, `node:http` request, dan WebSocket muncul di Network tab persis seperti saat debugging aplikasi browser. Ini adalah fitur yang saya rasa akan sangat mengubah cara developer profile aplikasi server-side mereka.

Plus, built-in CPU profiler dengan output format yang beragam: `.cpuprofile` untuk Chrome DevTools, interactive SVG flamegraph yang bisa dibuka di browser tanpa tool tambahan, dan Markdown report dengan daftar hot functions — yang terakhir ini sangat berguna untuk quick triage langsung dari terminal.

## Apa Artinya bagi Developer Web

Sebagai developer yang menggunakan React dan Node.js sehari-hari, saya melihat beberapa implikasi praktis dari evolusi Deno.

**Pertama, alternatif yang viable.** Selama ini Node.js adalah pilihan default yang hampir tidak pernah dipertanyakan di dunia server-side JavaScript. Tapi dengan Deno yang kompatibel 76% dengan Node.js test suite dan cold install yang 3.66x lebih cepat, kita punya runtime alternatif yang sudah bisa dipertimbangkan untuk production — bukan hanya untuk proyek sampingan.

**Kedua, toolchain yang menyederhanakan.** Deno sekarang mencakup runtime, package manager, test runner, linter, formatter, bundler, version bumper, audit tool, dan profiler — semua dalam satu binary tanpa konfigurasi. Bandingkan dengan ekosistem Node.js yang butuh `node`, `npm`/`pnpm`, `jest`/`vitest`, `eslint`, `prettier`, dan mungkin `webpack`/`vite`, masing-masing dengan config file sendiri. Dalam dunia yang sudah complicated, kesederhanaan ini punya value.

**Ketiga, keamanan by design.** Permission model Deno — dimana program tidak bisa mengakses file system, network, atau environment variable tanpa explicit permission — adalah pendekatan yang lebih aman secara fundamental. Di era supply chain attack yang makin sering menarget npm ecosystem, ini bukan fitur kosmetik.

Tentu ada caveat. Ekosistem Deno masih jauh lebih kecil dari Node.js. Beberapa package npm yang bergantung pada native addon atau API Node yang sangat spesifik mungkin masih bermasalah. Dan sebagai engineering manager, saya juga perlu mempertimbangkan kurva belajar tim dan availability talent yang familiar dengan Deno di pasar Indonesia — yang saat ini masih sangat terbatas.

## Runtime Wars: Ke Mana Arahnya?

Lanskap JavaScript runtime tahun 2026 menarik untuk diamati. Kita punya tiga pemain utama:

- **Node.js** masih raja dengan ekosistem terbesar dan 17 tahun track record, tapi secara arsitektur membawa legacy decision dari 2009 yang semakin sulit diubah
- **Deno** yang bertumbuh pesat dengan pendekatan "secure by default," toolchain terintegrasi, dan sekarang kompatibilitas npm yang solid
- **Bun** yang sedang dalam transformasi pasca-akuisisi Anthropic — masa depannya bergantung pada seberapa dalam integrasinya dengan ecosystem AI

Yang menarik, ketiga runtime ini sekarang saling meminjam fitur. Node.js mengadopsi fitur yang awalnya populer di Deno (seperti permission model eksperimental dan native test runner). Deno membangun kompatibilitas npm yang masif. Bun berfokus pada kecepatan ekstrem dan integrasi dengan AI workflow.

Persaingan ini menguntungkan kita sebagai developer. Inovasi di satu runtime mendorong yang lain untuk berkembang. Dan dengan standar web API yang makin konvergen — `fetch()`, `WebSocket`, `crypto.subtle`, dan seterusnya — kode yang kita tulis makin portable antar runtime.

## Refleksi Penutup

Saya tidak sedang mengatakan semua orang harus migrasi ke Deno besok. Node.js tetap pilihan paling aman untuk production, dengan ekosistem yang sudah teruji dan komunitas yang massive. Tapi yang saya lihat dari Deno 2.8 adalah tanda bahwa alternatif itu sudah matang — bukan lagi eksperimen akhir pekan, tapi tools yang bisa dipertimbangkan serius untuk proyek baru.

Yang paling saya apresiasi dari pendekatan Deno adalah filosofi "batteries included" tanpa mengorbankan kesederhanaan. Dalam dunia frontend yang sudah complicated dengan build tools, bundler, dan konfigurasi yang tak berujung, ada sesuatu yang refreshing tentang runtime yang bisa menjalankan `deno run main.ts` tanpa setup apa pun — dan sekarang juga bisa menginstal package npm dengan kecepatan yang jauh lebih cepat.

Mungkin bukan saatnya meninggalkan Node.js. Tapi pasti saatnya untuk mencoba Deno di proyek sampingan berikutnya.
