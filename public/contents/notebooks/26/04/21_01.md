# WebAssembly: Bahasa Apapun, Jalan di Browser

![Ilustrasi pengembangan web modern](https://images.unsplash.com/photo-1555066931-4365d14bab8c?w=800&q=80)

JavaScript sudah jadi raja tunggal di browser selama lebih dari dua dekade. Kalau kamu mau bikin sesuatu yang jalan di web, kamu harus menulis JavaScript — itu yang selama ini jadi hukum tak tertulis. Tapi sejak 2017, ada pendatang baru yang perlahan mengubah aturan permainan: WebAssembly, atau yang lebih akrab disapa Wasm.

Bukan berarti JavaScript akan punah. Justru, WebAssembly hadir bukan untuk menggantikan, tapi untuk melengkapi. Dan cara kerjanya cukup radikal: bayangkan kamu bisa menulis kode dalam C++, Rust, Go, atau bahkan Python, lalu menjalankannya di browser dengan performa mendekati native.

## Apa Sebenarnya WebAssembly Itu?

WebAssembly adalah format instruksi biner yang dirancang untuk stack-based virtual machine. Dikembangkan secara kolaboratif oleh W3C bersama Mozilla, Microsoft, Google, dan Apple, Wasm pertama kali mencapai status Candidate Recommendation pada Maret 2017 dan menjadi rekomendasi resmi W3C pada Desember 2019.

Menurut Web Almanac 2025 yang diterbitkan oleh HTTP Archive, WebAssembly telah berevolusi dari sekadar alat optimasi web menjadi "high-performance, universal bytecode format." Pada Desember 2025, Wasm Version 3.0 resmi menjadi standar W3C, mensandardisasi beberapa fitur lanjutan seperti Garbage Collection (WasmGC), 64-bit address space, dan Multiple Memories.

Cara kerjanya secara sederhana: kamu menulis kode dalam bahasa pemrograman yang didukung, lalu kompilator menerjemahkan kode tersebut menjadi format biner `.wasm`. File biner inilah yang kemudian dijalankan oleh browser. Browser modern — Chrome, Firefox, Safari, Edge — semuanya sudah mendukung WebAssembly secara native.

Yang menarik, Wasm bukan bahasa pemrograman. Dia lebih tepat disebut sebagai *compilation target* — sesuatu yang menjadi tujuan akhir kompilasi dari berbagai bahasa. Analoginya mirip seperti bytecode di Java atau CLR di .NET, tapi dirancang khusus untuk environment web.

## Angka-Angka: Seberapa Banyak yang Sudah Pakai?

Web Almanac 2025 memberikan data yang cukup gamblang tentang adopsi WebAssembly di web. Berdasarkan HTTP Archive July 2025 crawl data, mereka menemukan modul WebAssembly di **0.35% situs desktop dan 0.28% situs mobile**, yang merepresentasikan sekitar **43.000 situs**.

Angka itu mungkin terdengar kecil. Tapi konteksnya penting: adopsi Wasm tumbuh dari hanya 0.04% di tahun 2021. Dan kalau kita lihat situs-situs paling populer, angkanya jauh lebih tinggi — di **top 1.000 website, adopsinya mencapai 2% di desktop dan 1.27% di mobile**. Ini masuk akal karena situs-situs top-tier inilah yang paling sering menghosting aplikasi kompleks yang membutuhkan Wasm.

Sementara itu, data dari Chrome Platform Status yang dikutip byteiota menunjukkan perspektif berbeda: WebAssembly digunakan di **4.5% web application yang dikunjungi user Chrome** di 2025, tumbuh **28% year-over-year**. Ada proyeksi industri yang menargetkan 50% adopsi di 2030 — meski tentu angka proyeksi harus dibaca dengan skeptisisme sehat.

Kenapa ada dua angka yang berbeda (0.35% vs 4.5%)? Karena cara mengukurnya beda. HTTP Archive menghitung persentase dari total situs yang di-crawl, sementara Chrome Platform Status menghitung persentase dari web app yang *dikunjungi* user — artinya situs-situs besar yang pakai Wasm mendapat bobot lebih besar karena lebih sering dikunjungi.

## Kenapa WebAssembly Penting?

Performa. Itu jawaban singkatnya.

JavaScript adalah bahasa yang dinamis, JIT-compiled, dan tidak dirancang untuk komputasi berat. Ketika kamu perlu memproses video, menjalankan simulasi fisika, mengolah gambar dalam skala besar, atau melakukan enkripsi kompleks di browser, JavaScript cepat menemui batasnya.

WebAssembly mengatasi hal ini karena kode yang dikompilasi ke Wasm sudah berada dalam format yang hampir mendekati machine code. Tidak ada lagi parsing, tidak ada lagi proses JIT yang membutuhkan waktu warm-up.

Sebuah studi dari benchmarkingwasm.github.io yang melakukan pengujian sistematis menemukan bahwa **pada input berukuran kecil, WebAssembly lebih cepat dari JavaScript di 97.6% benchmark**. Angka itu turun menjadi 95.1% untuk input berukuran kecil-sedang. Studi ini dipublikasikan di IEEE Xplore sebagai perbandingan sistematis performa Wasm vs JavaScript.

Robert Aboukhalil, penulis buku "Level Up With WebAssembly", bahkan mendokumentasikan case study di Smashing Magazine di mana migrasi kalkulasi JavaScript ke Wasm menghasilkan **peningkatan kecepatan hingga 20x** untuk sebuah tool analisis data web.

## Dunia Nyata: Siapa yang Sudah Pakai?

Adopsi WebAssembly bukan lagi sekadar eksperimen. Beberapa produk dan perusahaan besar sudah mengandalkannya.

**Figma**, tool desain UI yang sangat populer, menggunakan Wasm untuk menjalankan engine render C++ mereka langsung di browser. Menurut blog post resmi Figma, migrasi dari asm.js ke WebAssembly menghasilkan **peningkatan performa load time hingga 3x**. Produk mereka yang ditulis dalam C++ bisa dikompilasi langsung ke Wasm, membuat Figma menjadi contoh sempurna kekuatan format ini.

**AutoDesk** menggunakannya untuk menjalankan **AutoCAD di browser** — aplikasi CAD profesional yang selama ini hanya bisa jalan di desktop. Platform.uno dalam laporan "State of WebAssembly 2024 and 2025" menyebutkan bahwa setiap tahun semakin banyak aplikasi komersial yang mengadopsi Wasm, dari video conferencing seperti **Zoom dan Google Meet**, software desain seperti Figma dan AutoCAD web, hingga social media platform.

**Adobe Photoshop web version** juga jadi contoh nyata. Web Almanac 2025 mencatat bahwa binary Wasm berukuran besar di percentile ke-90 (381 KB desktop, 316 KB mobile) biasanya merepresentasikan "full desktop-grade applications ported to the web — such as Adobe Photoshop or Google Earth — compiled from heavier languages like C++ or C#."

Fermyon, perusahaan yang baru diakuisisi Akamai untuk edge computing, menulis di blog mereka: "If you use Microsoft's online Office.com suite or the popular Figma graphics tool, you've already used Wasm in production. If you've used a device to watch streaming services like BBC or Amazon Prime, you've already used Wasm in production."

## Dari Browser ke Server: WASI dan Component Model

Di sinilah ceritanya jadi lebih menarik. WebAssembly awalnya dirancang untuk browser, tapi komunitas segera menyadari bahwa format biner portabel ini punya potensi jauh lebih besar.

Masuklah **WASI** — WebAssembly System Interface. WASI adalah spesifikasi yang mendefinisikan bagaimana modul Wasm berinteraksi dengan sistem operasi: akses file, jaringan, environment variable, dan lain-lain. Dengan WASI, WebAssembly bisa berjalan di server, di command line, di edge computing — di mana saja, bukan hanya di browser.

Artikel Java Code Geeks April 2026 memberikan diagnosis jujur: "The Component Model shipped. WASI Preview 2 is stable. Wasmtime hit 30+. Browser WASM is genuinely mature. And yet server-side Wasm keeps stalling on the same gaps." Threading disebut sebagai "the biggest missing primitive for compute-heavy server workloads."

Tapi meski begitu, beberapa use case server-side sudah berjalan:

- **Cloudflare Workers** memproses miliaran invokasi harian dengan cold start sub-milidetik menggunakan Wasm. Fermyon mencatat bahwa Fastly Compute@Edge juga berjalan di atas WebAssembly dengan sekitar 10.000 user.
- **Shopify** mengumumkan sistem plugin berbasis WebAssembly sejak 2020, memungkinkan merchant menulis custom logic dalam berbagai bahasa yang dikompilasi ke Wasm.
- **Docker** sekarang punya native Wasm support, menjadikan Wasm sebagai alternatif container yang lebih ringan.
- **Akamai mengakuisisi Fermyon** untuk edge computing, menandakan betapa seriusnya industri melihat Wasm sebagai infrastruktur cloud masa depan.

Gothar Tech dalam artikel mereka "The Five-Millisecond Cloud" merangkum: "WebAssembly workloads are running at planetary scale on Cloudflare Workers — millions of functions, sub-10ms cold starts."

## WebAssembly dan React Developer

Kalau kamu sehari-hari bekerja dengan React dan Node.js, mungkin terasa WebAssembly agak jauh dari rutinitas kamu. Tapi kenyataannya, ekosistem JavaScript mulai dipenuhi integrasi Wasm di balik layar.

Beberapa tool populer yang sudah memanfaatkan Wasm:

- **esbuild** — bundler yang sangat cepat, ditulis dalam Go dan dikompilasi ke Wasm untuk versi browser dan Node.js
- **SWC** — compiler JavaScript/TypeScript yang ditulis dalam Rust, dengan binding Wasm
- **SQL.js** — SQLite yang dikompilasi ke Wasm, bisa jalan penuh di browser tanpa backend

Di sisi Next.js, Vercel sudah mendukung Edge Runtime yang bisa menjalankan modul Wasm. Artinya, kamu bisa menulis middleware atau API route yang memanggil fungsi Wasm secara langsung.

Dalam pengalaman saya sendiri, ketika berhadapan dengan kebutuhan pemrosesan file besar di sisi client — misalnya parsing CSV ratusan megabyte atau mengolah gambar — pendekatan Wasm memberikan perbedaan performa yang signifikan dibanding murni JavaScript.

## Tantangan yang Masih Ada

WebAssembly bukan tanpa kelemahan, dan penting untuk memahami batasannya.

**Tidak bisa akses DOM langsung.** Wasm tidak punya akses langsung ke Document Object Model. Untuk memanipulasi elemen HTML, kamu tetap perlu JavaScript sebagai perantara. Ini menambah overhead komunikasi.

**Server-side masih "almost ready".** Artikel Java Code Geeks April 2026 menyoroti ini dengan judul yang berbicara banyak: "Three Years of Almost Ready." Component Model sudah ship, WASI Preview 2 sudah stabil, tapi threading — fitur krusial untuk server workload berat — masih belum ada.

**Debugging lebih sulit.** Meskipun browser developer tool sudah mendukung Wasm debugging, pengalamannya belum semulus debug JavaScript. Source map tersedia tapi tidak selalu sempurna.

**Ukuran file bisa besar.** Web Almanac 2025 mencatat bahwa di percentile ke-90, ukuran binary Wasm mencapai 381 KB di desktop. Binary besar ini biasanya berasal dari aplikasi desktop-grade yang di-port ke web.

**Ekosistem masih berkembang.** Tidak seperti JavaScript yang punya npm dengan jutaan package, library Wasm yang bisa langsung dipakai masih terbatas.

## Masa Depan: Ke Mana Arahnya?

Meski tantangan masih ada, beberapa perkembangan menjanjikan:

- **WasmGC** sudah distandardisasi di Wasm 3.0, membuka jalan bagi bahasa seperti Kotlin, Dart (Flutter), dan Java untuk dikompilasi ke Wasm lebih efisien. Platform.uno mencatat bahwa Dart/Flutter Web sekarang menggunakan WasmGC sebagai target kompilasi utamanya.
- **Component Model** memungkinkan modul Wasm dari bahasa berbeda saling berkomunikasi. Bayangkan modul Rust memanggil modul Python, keduanya dalam format Wasm.
- **WASI Preview 2** sudah stabil, membuat server-side Wasm semakin viable.

Ada juga kritik yang perlu didengar. Artikel Java Code Geeks mengingatkan bahwa narasi "Wasm akan menggantikan container" sudah berumur tiga tahun tanpa bukti kuat di lapangan. Cold start memang cepat, tapi banyak workload server tidak peduli dengan cold start — yang mereka butuhkan adalah threading, observability, dan ecosystem maturity.

## Harus Mulai Belajar WebAssembly Sekarang?

Jawabannya bergantung pada konteksnya.

Kalau kamu membangun aplikasi web yang sebagian besar adalah CRUD, form, dan dashboard standar, JavaScript (atau TypeScript) sudah lebih dari cukup. Tidak ada kebutuhan mendesak untuk beralih ke Wasm.

Tapi kalau kamu mulai menyentuh area seperti pemrosesan data besar di browser, image/video processing, simulasi, game, atau cryptography, WebAssembly bukan lagi nice-to-have — ia menjadi kebutuhan teknis.

Untuk mulai, Rust adalah pilihan paling populer berkat toolchain `wasm-pack` yang matang dan ekosistem `wasm-bindgen`. Tapi kalau kamu sudah familiar dengan Go atau C++, kompilator ke Wasm juga sudah tersedia.

Langkah paling realistis: gunakan tool yang sudah memanfaatkan Wasm di balik layar. Coba esbuild, mainkan dengan SQL.js, atau eksplor image processing library berbasis Wasm. Dari situ, kamu akan mulai memahami kapan dan di mana Wasm memberikan nilai tambah yang nyata.

WebAssembly bukan hype yang akan hilang besok. Dia adalah perubahan fundamental dalam cara kita memikirkan eksekusi kode — di web, dan semakin lama, di mana saja.
